# Smart Shield — API Gateway (Go)

The public GraphQL API for the Smart Shield trading platform, built as a
**modular monolith**: one deployable binary, internally split into independent
feature modules (`user`, `portfolio`) that could each become their own service
later without a rewrite.

- **Transport:** GraphQL (gqlgen, schema-first) over HTTP (gin)
- **Storage:** Postgres via `pgx` + `sqlc` (type-safe queries), Redis for caching
- **Architecture:** clean / hexagonal — dependencies point inward toward `domain`

> The Python AI/ML service is an external gRPC dependency and is **not** part of
> this repo. Nothing here calls or imports it.

---

## Layering rules (the important part)

Every module is split into four layers. **Imports only ever point inward:**

```
interfaces (GraphQL resolvers, HTTP)      ─┐
infrastructure (postgres, redis, billing) ─┼─► usecase ─► domain
platform (config, logger, db pool, jwt)   ─┘
```

| Layer | Package(s) | May import | May NOT import |
|-------|-----------|------------|----------------|
| **domain** | `internal/<mod>/domain` | stdlib, `decimal`, `apperror` | usecase, repo, gqlgen, pgx, gin |
| **usecase** | `internal/<mod>/usecase` | its `domain`, `apperror` | pgx, sql, gqlgen, gin, gorm, bcrypt/jwt directly |
| **repository (infra)** | `internal/<mod>/repository/...` | its `domain`, `apperror`, pgx, sqlc `db` | usecase, gqlgen |
| **interfaces** | `internal/graphql/...` | usecases, `domain`, gqlgen | pgx, sql |
| **platform** | `internal/platform/...` | stdlib + libs | any feature module's `domain` |

Concretely, this is enforced by discipline + review, and made easy by the fact
that **the domain defines interfaces (ports)** and infrastructure implements
them:

- `domain.UserRepository` (port) ⟵ implemented by `repository/postgres.UserRepo`
- `domain.PasswordHasher` / `domain.TokenIssuer` ⟵ `platform/security`
- `domain.BillingProvider` ⟵ `repository/billing.StripeStub`

Usecases receive these as interfaces via constructors, so they never see pgx,
bcrypt, or jwt — and are unit-testable with plain in-memory fakes (see
`*_test.go`).

## Folder structure

```
services/api/
├── cmd/server/main.go          # composition root — the ONLY place that wires concretes
├── api/graphql/schema/         # schema-first .graphqls, one file per module
├── internal/
│   ├── platform/               # cross-cutting infra, injected everywhere
│   │   ├── config/  logger/  postgres/  redis/  security/  apperror/  ctxkit/
│   ├── graphql/                # interfaces layer (gqlgen)
│   │   ├── generated/ model/   # ⚙️ GENERATED by gqlgen — do not edit
│   │   ├── scalar/             # Decimal scalar binding
│   │   ├── resolver/           # thin resolvers + domain↔model mappers
│   │   ├── dataloader/         # per-request batching (Trade.user N+1 guard)
│   │   ├── directives.go errors.go middleware.go server.go
│   ├── user/                   # ===== USER & AUTH bounded context =====
│   │   ├── domain/             # User, Profile, RefreshToken, ports
│   │   ├── usecase/            # auth, profile, subscription
│   │   └── repository/
│   │       ├── postgres/       # sqlc queries + adapter; db/ is ⚙️ GENERATED
│   │       └── billing/        # Stripe stub (BillingProvider)
│   └── portfolio/              # ===== PORTFOLIO & TRADE JOURNAL context =====
│       ├── domain/ usecase/ repository/postgres/
├── migrations/                 # golang-migrate NNNNNN_*.up.sql / .down.sql
├── deployments/                # Dockerfile + docker-compose (pg + redis)
├── gqlgen.yml  sqlc.yaml  .golangci.yml  .env.example  Makefile
```

### How the two modules share the DB pool without coupling

`main.go` builds **one** `*pgxpool.Pool` (`platform/postgres`) and passes it into
each module's repository constructor. The modules share the *connection*, never
each other's code. The only cross-module link is the `Trade.user` GraphQL field,
resolved at the interfaces layer via a dataloader that calls the user module's
usecase — not its internals.

## Data layer: why sqlc (and the tradeoff)

We use **sqlc**: you write plain SQL in `queries.sql`, and `make generate`
produces type-safe Go in a private `db/` package that only the repository
adapter imports.

- **Pro:** real SQL (no ORM surprises), compile-time-checked columns/types,
  native pgx, generated code stays behind the domain port.
- **Con:** sqlc is weak at *dynamic* queries. We handle optional filters with the
  **NULL-guard idiom** (see `portfolio/.../queries.sql`, `ListTradesByUser`) so a
  single static query covers every filter combination. For genuinely dynamic SQL
  you'd drop to hand-written pgx on the same pool — none is needed yet.

Alternative considered: hand-written repos. Rejected — more boilerplate and
easy-to-introduce scan bugs, with no offsetting benefit here.

## First-time setup

Requires: Go 1.25+, Docker. Install the codegen/dev tools once:

```bash
go install github.com/sqlc-dev/sqlc/cmd/sqlc@latest
go install -tags 'postgres' github.com/golang-migrate/migrate/v4/cmd/migrate@latest
go install github.com/golangci/golangci-lint/cmd/golangci-lint@latest
```

Then:

```bash
cd services/api
cp .env.example .env
go mod tidy            # populate go.sum / indirect deps (needs network)
make generate          # ⚙️ gqlgen + sqlc — creates generated/, model/, db/ packages
make docker-up         # start Postgres + Redis
make migrate-up        # apply migrations
make run               # http://localhost:8080  (playground at /)
```

> **Codegen is required before the first build.** These packages are produced by
> `make generate` and are git-ignored-by-convention here (commit them if you
> prefer): `internal/graphql/generated`, `internal/graphql/model`,
> `internal/user/repository/postgres/db`, `internal/portfolio/repository/postgres/db`.

## Testing

```bash
make test              # unit tests (table-driven, mocked repos) — fast, no Docker
make test-integration  # per-module testcontainers tests against real Postgres
```

- **Unit:** `internal/user/usecase/*_test.go`, `internal/portfolio/usecase/*_test.go`
  — pure logic against in-memory fakes of the domain ports.
- **Integration:** `.../repository/postgres/integration_test.go` — spin up
  Postgres via testcontainers, run migrations, exercise the real adapter. Gated
  by `testing.Short()`, so `make test` skips them.

## Errors → GraphQL

Usecases return typed `apperror.Error` values (`NOT_FOUND`, `UNAUTHENTICATED`,
`FORBIDDEN`, `INVALID_INPUT`, `CONFLICT`, `INTERNAL`). The resolver boundary's
`ErrorPresenter` (`internal/graphql/errors.go`) maps the code to
`extensions.code` and hides internal-error details (logging the cause). Clients
switch on `extensions.code`, never on message text.

## Auth model

- **Access token:** short-lived JWT (15m), sent as `Authorization: Bearer <token>`.
- **Refresh token:** opaque random string; only its SHA-256 hash is stored, with
  rotation on every `refreshToken` call and revocation on `logout`.
- HTTP middleware does **optional** auth (attaches identity if a valid token is
  present); the schema's `@auth` directive **enforces** it per field, so
  `login`/`register` stay public.

## Adding a new module (e.g. `learning`, `notifications`)

1. `internal/learning/domain` — entities + port interfaces (no framework imports).
2. `internal/learning/usecase` — business logic against those ports.
3. `internal/learning/repository/postgres` — `queries.sql` + adapter; add a
   `sql:` block to `sqlc.yaml`.
4. `api/graphql/schema/learning.graphqls` — `extend type Query/Mutation`.
5. Add resolver methods in `internal/graphql/resolver/learning.resolvers.go`
   (thin) + mappers; add the usecases to the `Resolver` struct.
6. Wire it in `cmd/server/main.go`.
7. `make generate`.

Same shape every time — that's the point.

## Notes / assumptions for this pass

- HTTP framework kept as **gin**; password hashing kept as **bcrypt** (spec allowed either).
- `UpdateTrade` cannot re-open a closed trade by clearing `exitPrice` (documented MVP limitation).
- Subscription changes apply immediately via the **stub** billing provider; with
  real Stripe the tier would flip on a webhook instead (the usecase already talks
  only to the `BillingProvider` port, so that swap touches no business logic).
- `Role` (user/admin, authz) and `SubscriptionTier` (billing) are intentionally
  separate concerns.
